AI 에이전트 안전 배포 ​

Claude Code와 Agent SDK는 코드 실행, 파일 접근, 외부 서비스 연동 등 강력한 기능을 제공하는 도구입니다. 이러한 기능을 갖춘 모든 도구와 마찬가지로, 사려 깊은 배포를 통해 이점을 누리면서도 적절한 통제를 유지할 수 있습니다.

전통적인 소프트웨어가 미리 정해진 코드 경로를 따르는 것과 달리, 이 도구들은 컨텍스트와 목표에 따라 동적으로 행동을 생성합니다. 이러한 유연성이 도구를 유용하게 만들지만, 동시에 처리하는 콘텐츠(파일, 웹페이지, 사용자 입력)에 의해 동작이 영향을 받을 수 있다는 것을 의미합니다. 이를 프롬프트 인젝션(prompt injection)이라고 합니다. 예를 들어, 저장소의 README에 비정상적인 지시사항이 포함되어 있다면 Claude Code가 운영자가 의도하지 않은 방식으로 그 내용을 행동에 반영할 수 있습니다. 이 가이드에서는 이러한 위험을 줄이는 실용적인 방법을 다룹니다.

좋은 소식은 에이전트 배포를 보안하기 위해 특수한 인프라가 필요하지 않다는 것입니다. 반신뢰(semi-trusted) 코드 실행에 적용되는 원칙들, 즉 격리, 최소 권한, 심층 방어가 여기서도 동일하게 적용됩니다. Claude Code에는 일반적인 보안 문제를 해결하는 여러 보안 기능이 포함되어 있으며, 이 가이드에서는 이러한 기능들과 함께 추가적인 강화 옵션을 안내합니다.

모든 배포가 최고 수준의 보안을 필요로 하는 것은 아닙니다. 개인 노트북에서 Claude Code를 실행하는 개발자와 멀티 테넌트 환경에서 고객 데이터를 처리하는 기업은 서로 다른 요구사항을 가집니다. 이 가이드는 Claude Code의 내장 보안 기능부터 강화된 프로덕션 아키텍처까지 다양한 옵션을 제시하므로 상황에 맞는 방법을 선택할 수 있습니다.

위협 모델 ​

에이전트는 프롬프트 인젝션(처리하는 콘텐츠에 포함된 지시사항)이나 모델 오류로 인해 의도하지 않은 행동을 할 수 있습니다. Claude 모델은 이에 저항하도록 설계되어 있으며, 모델 카드 분석에 따르면 Claude Opus 4.6은 현재 사용 가능한 가장 강인한 프런티어 모델입니다.

그럼에도 불구하고 심층 방어는 여전히 좋은 실천 방안입니다. 예를 들어 에이전트가 고객 데이터를 외부 서버로 전송하도록 지시하는 악성 파일을 처리하더라도, 네트워크 제어를 통해 해당 요청을 완전히 차단할 수 있습니다.

내장 보안 기능 ​

Claude Code에는 일반적인 보안 문제를 해결하는 여러 보안 기능이 포함되어 있습니다. 자세한 내용은 보안 문서를 참조하세요.

  • 권한 시스템: 모든 도구와 bash 명령을 허용, 차단, 또는 사용자 승인 요청으로 구성할 수 있습니다. 글로브 패턴을 사용해 "모든 npm 명령 허용" 또는 "sudo가 포함된 명령 차단"과 같은 규칙을 만들 수 있습니다. 조직은 모든 사용자에게 적용되는 정책을 설정할 수 있습니다. 권한 설정을 참조하세요.
  • 정적 분석: bash 명령 실행 전, Claude Code는 잠재적으로 위험한 작업을 식별하기 위해 정적 분석을 실행합니다. 시스템 파일을 수정하거나 민감한 디렉토리에 접근하는 명령은 플래그가 지정되어 명시적인 사용자 승인이 필요합니다.
  • 웹 검색 요약: 검색 결과는 원시 콘텐츠를 컨텍스트에 직접 전달하는 대신 요약하여 제공함으로써, 악성 웹 콘텐츠에 의한 프롬프트 인젝션 위험을 줄입니다.
  • 샌드박스 모드: Bash 명령은 파일 시스템과 네트워크 접근을 제한하는 샌드박스 환경에서 실행될 수 있습니다. 자세한 내용은 샌드박스 문서를 참조하세요.

보안 원칙 ​

Claude Code의 기본 설정 이상의 추가적인 강화가 필요한 배포를 위해, 다음 원칙들이 사용 가능한 옵션을 안내합니다.

보안 경계 ​

보안 경계는 서로 다른 신뢰 수준을 가진 구성 요소들을 분리합니다. 고보안 배포에서는 민감한 리소스(예: 자격 증명)를 에이전트가 포함된 경계 밖에 둘 수 있습니다. 에이전트 환경에서 문제가 발생하더라도, 해당 경계 외부의 리소스는 보호된 상태로 유지됩니다.

예를 들어, 에이전트에 API 키에 직접 접근 권한을 부여하는 대신, 에이전트 환경 외부에서 프록시를 실행하여 요청에 키를 삽입할 수 있습니다. 에이전트는 API 호출을 할 수 있지만 자격 증명 자체는 볼 수 없습니다. 이 패턴은 멀티 테넌트 배포나 신뢰할 수 없는 콘텐츠를 처리할 때 유용합니다.

최소 권한 ​

필요한 경우, 에이전트를 특정 작업에 필요한 기능만으로 제한할 수 있습니다:

리소스제한 옵션
파일 시스템필요한 디렉토리만 마운트, 읽기 전용 선호
네트워크프록시를 통해 특정 엔드포인트로 제한
자격 증명직접 노출 대신 프록시를 통해 주입
시스템 기능컨테이너에서 Linux 기능 제거

심층 방어 ​

고보안 환경에서는 여러 제어를 겹겹이 적용하여 추가적인 보호를 제공합니다. 옵션은 다음과 같습니다:

  • 컨테이너 격리
  • 네트워크 제한
  • 파일 시스템 제어
  • 프록시에서의 요청 검증

적합한 조합은 위협 모델과 운영 요구사항에 따라 다릅니다.

격리 기술 ​

격리 기술마다 보안 강도, 성능, 운영 복잡성 간에 서로 다른 트레이드오프가 있습니다.

INFO

이 모든 구성에서 Claude Code(또는 Agent SDK 애플리케이션)는 격리 경계(샌드박스, 컨테이너, VM) 내부에서 실행됩니다. 아래에서 설명하는 보안 제어는 에이전트가 해당 경계 내에서 접근할 수 있는 것을 제한합니다.

샌드박스 런타임 ​

컨테이너 없이 가벼운 격리를 위해, sandbox-runtime은 OS 수준에서 파일 시스템과 네트워크 제한을 적용합니다.

주요 장점은 단순성입니다. Docker 구성, 컨테이너 이미지, 네트워킹 설정이 필요 없습니다. 프록시와 파일 시스템 제한이 내장되어 있습니다. 허용된 도메인과 경로를 지정하는 설정 파일만 제공하면 됩니다.

작동 방식:

  • 파일 시스템: OS 기본 기능(bubblewrap on Linux, sandbox-exec on macOS)을 사용하여 구성된 경로에 대한 읽기/쓰기 접근을 제한합니다.
  • 네트워크: 네트워크 네임스페이스(Linux) 제거 또는 Seatbelt 프로파일(macOS)을 사용하여 내장 프록시를 통해 네트워크 트래픽을 라우팅합니다.
  • 구성: 도메인과 파일 시스템 경로에 대한 JSON 기반 허용 목록

설정:

bash
npm install @anthropic-ai/sandbox-runtime

그런 다음 허용된 경로와 도메인을 지정하는 구성 파일을 만드세요.

보안 고려사항:

  1. 동일 호스트 커널: VM과 달리, 샌드박스 프로세스는 호스트 커널을 공유합니다. 이론적으로 커널 취약점으로 인해 탈출이 가능할 수 있습니다. 일부 위협 모델에서는 이것이 허용될 수 있지만, 커널 수준의 격리가 필요한 경우 gVisor 또는 별도의 VM을 사용하세요.

  2. TLS 검사 없음: 프록시는 도메인을 허용 목록에 올리지만 암호화된 트래픽을 검사하지 않습니다. 에이전트가 허용된 도메인에 대한 허용적인 자격 증명을 가지고 있다면, 해당 도메인을 통해 다른 네트워크 요청을 트리거하거나 데이터를 유출하는 것이 불가능한지 확인하세요.

많은 개인 개발자 및 CI/CD 사용 사례에서, sandbox-runtime은 최소한의 설정으로 보안 수준을 크게 높입니다. 아래 섹션에서는 더 강력한 격리가 필요한 배포를 위한 컨테이너와 VM을 다룹니다.

컨테이너 ​

컨테이너는 Linux 네임스페이스를 통해 격리를 제공합니다. 각 컨테이너는 호스트 커널을 공유하면서도 파일 시스템, 프로세스 트리, 네트워크 스택에 대한 독립적인 뷰를 가집니다.

보안이 강화된 컨테이너 구성은 다음과 같을 수 있습니다:

bash
docker run \
  --cap-drop ALL \
  --security-opt no-new-privileges \
  --security-opt seccomp=/path/to/seccomp-profile.json \
  --read-only \
  --tmpfs /tmp:rw,noexec,nosuid,size=100m \
  --tmpfs /home/agent:rw,noexec,nosuid,size=500m \
  --network none \
  --memory 2g \
  --cpus 2 \
  --pids-limit 100 \
  --user 1000:1000 \
  -v /path/to/code:/workspace:ro \
  -v /var/run/proxy.sock:/var/run/proxy.sock:ro \
  agent-image

각 옵션의 역할:

옵션목적
--cap-drop ALL권한 상승을 가능하게 하는 NET_ADMIN, SYS_ADMIN 등의 Linux 기능을 제거합니다
--security-opt no-new-privileges프로세스가 setuid 바이너리를 통해 권한을 얻는 것을 방지합니다
--security-opt seccomp=...사용 가능한 시스템 콜을 제한합니다. Docker 기본값은 약 44개를 차단하며, 사용자 정의 프로파일로 더 많이 차단할 수 있습니다
--read-only컨테이너의 루트 파일 시스템을 불변으로 만들어 에이전트가 변경 사항을 영구적으로 저장하는 것을 방지합니다
--tmpfs /tmp:...컨테이너 중지 시 삭제되는 쓰기 가능한 임시 디렉토리를 제공합니다
--network none모든 네트워크 인터페이스를 제거합니다. 에이전트는 아래에 마운트된 Unix 소켓을 통해 통신합니다
--memory 2g리소스 고갈을 방지하기 위해 메모리 사용량을 제한합니다
--pids-limit 100fork bomb을 방지하기 위해 프로세스 수를 제한합니다
--user 1000:1000루트가 아닌 사용자로 실행합니다
-v ...:/workspace:ro코드를 읽기 전용으로 마운트하여 에이전트가 분석할 수 있지만 수정할 수 없게 합니다. ~/.ssh, ~/.aws, ~/.config 등 민감한 호스트 디렉토리를 마운트하지 마세요
-v .../proxy.sock:...컨테이너 외부에서 실행 중인 프록시에 연결된 Unix 소켓을 마운트합니다 (아래 참조)

Unix 소켓 아키텍처:

--network none을 사용하면 컨테이너에 네트워크 인터페이스가 전혀 없습니다. 에이전트가 외부 세계에 도달하는 유일한 방법은 마운트된 Unix 소켓을 통해서이며, 이는 호스트에서 실행 중인 프록시에 연결됩니다. 이 프록시는 도메인 허용 목록을 적용하고, 자격 증명을 주입하며, 모든 트래픽을 기록할 수 있습니다.

이것은 sandbox-runtime이 사용하는 것과 동일한 아키텍처입니다. 에이전트가 프롬프트 인젝션으로 인해 손상되더라도, 임의의 서버로 데이터를 유출할 수 없습니다. 오직 프록시를 통해서만 통신할 수 있으며, 프록시가 도달 가능한 도메인을 제어합니다. 자세한 내용은 Claude Code 샌드박스 블로그 포스트를 참조하세요.

추가 강화 옵션:

옵션목적
--userns-remap컨테이너 루트를 권한이 없는 호스트 사용자에 매핑합니다. 데몬 구성이 필요하지만 컨테이너 탈출 시 피해를 제한합니다
--ipc private교차 컨테이너 공격을 방지하기 위해 프로세스 간 통신을 격리합니다

gVisor ​

표준 컨테이너는 호스트 커널을 공유합니다. 컨테이너 내부의 코드가 시스템 콜을 만들면, 그것은 호스트를 실행하는 동일한 커널로 직접 전달됩니다. 이는 커널 취약점으로 인해 컨테이너 탈출이 가능할 수 있음을 의미합니다. gVisor는 시스템 콜을 호스트 커널에 도달하기 전에 사용자 공간에서 가로채어, 실제 커널을 포함하지 않고도 대부분의 시스템 콜을 처리하는 자체 호환성 레이어를 구현하여 이 문제를 해결합니다.

에이전트가 악성 코드를 실행하더라도(프롬프트 인젝션으로 인해), 해당 코드는 컨테이너 내에서 실행되며 커널 익스플로잇을 시도할 수 있습니다. gVisor를 사용하면 공격 표면이 훨씬 작아집니다. 악성 코드는 먼저 gVisor의 사용자 공간 구현을 익스플로잇해야 하며, 실제 커널에 대한 접근이 제한됩니다.

Docker와 함께 gVisor를 사용하려면 runsc 런타임을 설치하고 데몬을 구성하세요:

json
// /etc/docker/daemon.json
{
  "runtimes": {
    "runsc": {
      "path": "/usr/local/bin/runsc"
    }
  }
}

그런 다음 다음으로 컨테이너를 실행하세요:

bash
docker run --runtime=runsc agent-image

성능 고려사항:

워크로드오버헤드
CPU 집약적 연산~0% (시스템 콜 가로채기 없음)
단순 시스템 콜~2배 느림
파일 I/O 집약적빈번한 open/close 패턴에서 최대 10~200배 느림

멀티 테넌트 환경이나 신뢰할 수 없는 콘텐츠를 처리할 때, 추가적인 격리는 종종 오버헤드를 감수할 가치가 있습니다.

가상 머신 ​

VM은 CPU 가상화 확장을 통해 하드웨어 수준의 격리를 제공합니다. 각 VM은 자체 커널을 실행하여 강력한 경계를 생성합니다. 게스트 커널의 취약점은 호스트를 직접 침해하지 않습니다. 그러나 VM이 gVisor와 같은 대안보다 자동으로 "더 안전한" 것은 아닙니다. VM 보안은 하이퍼바이저와 디바이스 에뮬레이션 코드에 크게 의존합니다.

Firecracker는 경량 마이크로VM 격리를 위해 설계되었습니다. 불필요한 디바이스 에뮬레이션을 제거하여 공격 표면을 줄이면서, 5 MiB 미만의 메모리 오버헤드로 125ms 이내에 VM을 부팅할 수 있습니다.

이 접근 방식에서 에이전트 VM은 외부 네트워크 인터페이스가 없습니다. 대신 vsock(가상 소켓)을 통해 통신합니다. 모든 트래픽은 vsock을 통해 호스트의 프록시로 라우팅되며, 프록시는 요청을 전달하기 전에 허용 목록을 적용하고 자격 증명을 주입합니다.

클라우드 배포 ​

클라우드 배포의 경우, 위의 격리 기술 중 하나를 클라우드 네이티브 네트워크 제어와 결합할 수 있습니다:

  1. 인터넷 게이트웨이 없는 프라이빗 서브넷에서 에이전트 컨테이너 실행
  2. 클라우드 방화벽 규칙(AWS Security Groups, GCP VPC 방화벽)을 구성하여 프록시를 제외한 모든 이그레스 차단
  3. 요청을 검증하고, 도메인 허용 목록을 적용하며, 자격 증명을 주입하고 외부 API로 전달하는 프록시 실행 (Envoy의 credential_injector 필터 사용 등)
  4. 에이전트의 서비스 계정에 최소한의 IAM 권한을 할당하고, 가능한 경우 민감한 접근은 프록시를 통해 라우팅
  5. 감사 목적으로 프록시에서 모든 트래픽 기록

자격 증명 관리 ​

에이전트는 API 호출, 저장소 접근, 클라우드 서비스 연동을 위해 자격 증명이 필요한 경우가 많습니다. 자격 증명 자체를 노출하지 않으면서 접근을 제공하는 것이 과제입니다.

프록시 패턴 ​

권장되는 접근 방식은 에이전트의 보안 경계 외부에서 프록시를 실행하여 나가는 요청에 자격 증명을 주입하는 것입니다. 에이전트는 자격 증명 없이 요청을 보내고, 프록시가 자격 증명을 추가한 후 목적지로 요청을 전달합니다.

이 패턴은 여러 가지 이점을 제공합니다:

  1. 에이전트가 실제 자격 증명을 볼 수 없습니다
  2. 프록시가 허용된 엔드포인트의 허용 목록을 적용할 수 있습니다
  3. 프록시가 감사를 위해 모든 요청을 기록할 수 있습니다
  4. 자격 증명이 각 에이전트에 분산되지 않고 한 곳에서 안전하게 저장됩니다

Claude Code에서 프록시 사용 구성 ​

Claude Code는 샘플링 요청을 프록시를 통해 라우팅하는 두 가지 방법을 지원합니다:

옵션 1: ANTHROPIC_BASE_URL (간단하지만 샘플링 API 요청에만 해당)

bash
export ANTHROPIC_BASE_URL="http://localhost:8080"

이렇게 하면 Claude Code와 Agent SDK가 Claude API 대신 프록시로 샘플링 요청을 보냅니다. 프록시는 일반 텍스트 HTTP 요청을 받아 검사하고 수정(자격 증명 주입 포함)한 후 실제 API로 전달합니다.

옵션 2: HTTP_PROXY / HTTPS_PROXY (시스템 전체)

bash
export HTTP_PROXY="http://localhost:8080"
export HTTPS_PROXY="http://localhost:8080"

Claude Code와 Agent SDK는 이 표준 환경 변수를 준수하여 모든 HTTP 트래픽을 프록시를 통해 라우팅합니다. HTTPS의 경우, 프록시는 암호화된 CONNECT 터널을 생성합니다. TLS 가로채기 없이는 요청 내용을 보거나 수정할 수 없습니다.

프록시 구현 ​

자체 프록시를 만들거나 기존 것을 사용할 수 있습니다:

  • Envoy Proxy: 인증 헤더 추가를 위한 credential_injector 필터가 있는 프로덕션 수준 프록시
  • mitmproxy: HTTPS 트래픽 검사 및 수정을 위한 TLS 종료 프록시
  • Squid: 접근 제어 목록이 있는 캐싱 프록시
  • LiteLLM: 자격 증명 주입 및 속도 제한이 있는 LLM 게이트웨이

다른 서비스의 자격 증명 ​

Claude API 샘플링 외에도, 에이전트는 종종 git 저장소, 데이터베이스, 내부 API 등 다른 서비스에 인증된 접근이 필요합니다. 두 가지 주요 접근 방식이 있습니다:

사용자 정의 도구 ​

에이전트의 보안 경계 외부에서 실행 중인 서비스로 요청을 라우팅하는 MCP 서버 또는 사용자 정의 도구를 통해 접근을 제공합니다. 에이전트가 도구를 호출하지만, 실제 인증된 요청은 외부에서 이루어집니다. 도구는 자격 증명을 주입하는 프록시를 호출합니다.

예를 들어, git MCP 서버는 에이전트의 명령을 받지만 호스트에서 실행 중인 git 프록시로 전달하고, 프록시가 원격 저장소에 연결하기 전에 인증을 추가합니다. 에이전트는 자격 증명을 볼 수 없습니다.

장점:

  • TLS 가로채기 불필요: 외부 서비스가 직접 인증된 요청을 만듭니다
  • 자격 증명이 외부에 유지됨: 에이전트는 도구 인터페이스만 볼 뿐, 기본 자격 증명은 볼 수 없습니다

트래픽 전달 ​

Claude API 호출의 경우, ANTHROPIC_BASE_URL을 사용하면 요청을 일반 텍스트에서 검사하고 수정할 수 있는 프록시로 라우팅할 수 있습니다. 그러나 다른 HTTPS 서비스(GitHub, npm 레지스트리, 내부 API)의 트래픽은 종종 종단 간 암호화됩니다. HTTP_PROXY를 통해 프록시로 라우팅하더라도, 프록시는 불투명한 TLS 터널만 볼 뿐 자격 증명을 주입할 수 없습니다.

사용자 정의 도구를 사용하지 않고 임의의 HTTPS 서비스에 대한 트래픽을 수정하려면 TLS 종료 프록시가 필요합니다. 이 프록시는 트래픽을 복호화하고, 검사하거나 수정한 후, 전달하기 전에 다시 암호화합니다. 이를 위해서는:

  1. 에이전트의 컨테이너 외부에서 프록시 실행
  2. 에이전트의 신뢰 저장소에 프록시의 CA 인증서 설치 (에이전트가 프록시의 인증서를 신뢰하도록)
  3. HTTP_PROXY/HTTPS_PROXY를 구성하여 프록시를 통해 트래픽 라우팅

이 접근 방식은 사용자 정의 도구 없이 HTTP 기반 서비스를 처리하지만, 인증서 관리의 복잡성이 추가됩니다.

모든 프로그램이 HTTP_PROXY/HTTPS_PROXY를 준수하는 것은 아닙니다. 대부분의 도구(curl, pip, npm, git)는 준수하지만, 일부는 이 변수를 무시하고 직접 연결할 수 있습니다. 예를 들어 Node.js fetch()는 기본적으로 이 변수를 무시합니다. Node 24+에서는 NODE_USE_ENV_PROXY=1을 설정하여 지원을 활성화할 수 있습니다. 포괄적인 범위를 위해, proxychains를 사용하여 네트워크 호출을 가로채거나, iptables를 구성하여 아웃바운드 트래픽을 투명 프록시로 리디렉션할 수 있습니다.

INFO

투명 프록시는 네트워크 수준에서 트래픽을 가로채므로 클라이언트가 프록시를 사용하도록 구성할 필요가 없습니다. 일반 프록시는 클라이언트가 명시적으로 연결하고 HTTP CONNECT 또는 SOCKS를 사용해야 합니다. 투명 프록시(투명 모드의 Squid나 mitmproxy 등)는 리디렉션된 원시 TCP 연결을 처리할 수 있습니다.

두 접근 방식 모두 TLS 종료 프록시와 신뢰할 수 있는 CA 인증서가 필요합니다. 다만 트래픽이 실제로 프록시에 도달하도록 보장하는 방식이 다릅니다.

파일 시스템 구성 ​

파일 시스템 제어는 에이전트가 읽고 쓸 수 있는 파일을 결정합니다.

읽기 전용 코드 마운트 ​

에이전트가 코드를 분석하되 수정할 필요가 없을 때, 디렉토리를 읽기 전용으로 마운트하세요:

bash
docker run -v /path/to/code:/workspace:ro agent-image

WARNING

코드 디렉토리에 대한 읽기 전용 접근도 자격 증명을 노출할 수 있습니다. 마운트 전에 제외하거나 정리해야 할 일반 파일들:

파일위험
.env, .env.localAPI 키, 데이터베이스 비밀번호, 시크릿
~/.git-credentials일반 텍스트의 Git 비밀번호/토큰
~/.aws/credentialsAWS 액세스 키
~/.config/gcloud/application_default_credentials.jsonGoogle Cloud ADC 토큰
~/.azure/Azure CLI 자격 증명
~/.docker/config.jsonDocker 레지스트리 인증 토큰
~/.kube/configKubernetes 클러스터 자격 증명
.npmrc, .pypirc패키지 레지스트리 토큰
*-service-account.jsonGCP 서비스 계정 키
*.pem, *.key개인 키

필요한 소스 파일만 복사하거나 .dockerignore 스타일 필터링 사용을 고려하세요.

쓰기 가능한 위치 ​

에이전트가 파일을 쓸 필요가 있는 경우, 변경 사항의 지속 여부에 따라 몇 가지 옵션이 있습니다:

컨테이너의 임시 작업 공간의 경우, 메모리에만 존재하고 컨테이너 중지 시 삭제되는 tmpfs 마운트를 사용하세요:

bash
docker run \
  --read-only \
  --tmpfs /tmp:rw,noexec,nosuid,size=100m \
  --tmpfs /workspace:rw,noexec,size=500m \
  agent-image

변경 사항을 지속하기 전에 검토하고 싶다면, 오버레이 파일 시스템을 사용하면 에이전트가 기본 파일을 수정하지 않고 쓸 수 있습니다. 변경 사항은 별도의 레이어에 저장되어 검사, 적용 또는 삭제할 수 있습니다. 완전히 지속적인 출력의 경우 전용 볼륨을 마운트하되 민감한 디렉토리와는 분리하세요.
