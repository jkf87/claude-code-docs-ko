# 샌드박싱

## 개요

Claude Code는 에이전트 실행을 위한 더 안전한 환경을 제공하고 지속적인 권한 프롬프트의 필요성을 줄이기 위해 네이티브 샌드박싱 기능을 갖추고 있습니다. bash 명령어마다 권한을 요청하는 대신, 샌드박싱은 Claude Code가 더 자유롭게 작업하되 위험은 줄일 수 있는 명확한 경계를 미리 설정합니다.

샌드박스 bash 도구는 OS 수준의 기본 요소를 사용하여 파일 시스템 격리와 네트워크 격리를 모두 적용합니다.

## 샌드박싱이 중요한 이유

기존의 권한 기반 보안 방식은 bash 명령어에 대해 지속적인 사용자 승인이 필요합니다. 이 방식은 제어권을 제공하지만 다음과 같은 문제가 발생할 수 있습니다:

* **승인 피로**: "승인"을 반복적으로 클릭하다 보면 실제로 무엇을 승인하는지 덜 주의를 기울이게 됩니다
* **생산성 저하**: 지속적인 방해가 개발 워크플로우를 느리게 합니다
* **자율성 제한**: 승인을 기다리는 동안 Claude Code가 효율적으로 작업할 수 없습니다

샌드박싱은 다음과 같은 방식으로 이러한 문제를 해결합니다:

1. **명확한 경계 정의**: Claude Code가 접근할 수 있는 디렉토리와 네트워크 호스트를 정확히 지정합니다
2. **권한 프롬프트 감소**: 샌드박스 내의 안전한 명령어는 승인이 필요 없습니다
3. **보안 유지**: 샌드박스 외부 리소스에 대한 접근 시도 시 즉각적인 알림을 발생시킵니다
4. **자율성 활성화**: Claude Code가 정의된 제한 내에서 더 독립적으로 실행될 수 있습니다

:::warning
효과적인 샌드박싱은 파일 시스템 격리와 네트워크 격리 **모두** 필요합니다. 네트워크 격리 없이는 침해된 에이전트가 SSH 키와 같은 민감한 파일을 외부로 유출할 수 있습니다. 파일 시스템 격리 없이는 침해된 에이전트가 시스템 리소스에 백도어를 심어 네트워크 접근 권한을 얻을 수 있습니다. 샌드박싱을 구성할 때 설정이 이러한 시스템에 우회 경로를 만들지 않도록 확인하는 것이 중요합니다.
:::

## 작동 방식

### 파일 시스템 격리

샌드박스 bash 도구는 파일 시스템 접근을 특정 디렉토리로 제한합니다:

* **기본 쓰기 동작**: 현재 작업 디렉토리와 하위 디렉토리에 대한 읽기 및 쓰기 접근
* **기본 읽기 동작**: 특정 거부된 디렉토리를 제외한 전체 컴퓨터에 대한 읽기 접근
* **차단된 접근**: 명시적 권한 없이 현재 작업 디렉토리 외부의 파일 수정 불가
* **설정 가능**: 설정을 통해 커스텀 허용/거부 경로 정의

설정에서 `sandbox.filesystem.allowWrite`를 사용하여 추가 경로에 대한 쓰기 접근 권한을 부여할 수 있습니다. 이러한 제한은 OS 수준에서 적용됩니다(macOS의 경우 Seatbelt, Linux의 경우 bubblewrap). 따라서 Claude의 파일 도구뿐만 아니라 `kubectl`, `terraform`, `npm` 등의 도구를 포함한 모든 서브프로세스 명령어에 적용됩니다.

### 네트워크 격리

네트워크 접근은 샌드박스 외부에서 실행되는 프록시 서버를 통해 제어됩니다:

* **도메인 제한**: 승인된 도메인만 접근 가능
* **사용자 확인**: 새 도메인 요청 시 권한 프롬프트 발생 ([`allowManagedDomainsOnly`](/settings#sandbox-settings)가 활성화된 경우 미허용 도메인은 자동으로 차단됨)
* **커스텀 프록시 지원**: 고급 사용자는 발신 트래픽에 커스텀 규칙 적용 가능
* **포괄적 적용**: 명령어에 의해 생성된 모든 스크립트, 프로그램, 서브프로세스에 제한 적용

### OS 수준 적용

샌드박스 bash 도구는 운영 체제 보안 기본 요소를 활용합니다:

* **macOS**: 샌드박스 적용을 위해 Seatbelt 사용
* **Linux**: 격리를 위해 [bubblewrap](https://github.com/containers/bubblewrap) 사용
* **WSL2**: Linux와 동일하게 bubblewrap 사용

WSL1은 bubblewrap이 WSL2에서만 사용 가능한 커널 기능을 필요로 하기 때문에 지원되지 않습니다.

이러한 OS 수준의 제한은 Claude Code 명령어에 의해 생성된 모든 자식 프로세스가 동일한 보안 경계를 상속받도록 보장합니다.

## 시작하기

### 사전 요구 사항

**macOS**에서는 내장 Seatbelt 프레임워크를 사용하여 샌드박싱이 바로 작동합니다.

**Linux 및 WSL2**에서는 먼저 필요한 패키지를 설치하세요:

::: code-group

```bash [Ubuntu/Debian]
sudo apt-get install bubblewrap socat
```

```bash [Fedora]
sudo dnf install bubblewrap socat
```

:::

### 샌드박싱 활성화

`/sandbox` 명령어를 실행하여 샌드박싱을 활성화할 수 있습니다:

```text
/sandbox
```

이 명령어는 샌드박스 모드를 선택할 수 있는 메뉴를 엽니다. 필요한 의존성이 없는 경우(Linux의 `bubblewrap` 또는 `socat` 등), 메뉴에 해당 플랫폼의 설치 지침이 표시됩니다.

기본적으로 샌드박스를 시작할 수 없는 경우(의존성 누락, 지원되지 않는 플랫폼, 또는 플랫폼 제한), Claude Code는 경고를 표시하고 샌드박싱 없이 명령어를 실행합니다. 이를 하드 실패로 처리하려면 [`sandbox.failIfUnavailable`](/settings#sandbox-settings)을 `true`로 설정하세요. 이 옵션은 샌드박싱을 보안 게이트로 요구하는 관리형 배포를 위한 것입니다.

### 샌드박스 모드

Claude Code는 두 가지 샌드박스 모드를 제공합니다:

**자동 허용 모드**: bash 명령어는 샌드박스 내에서 실행을 시도하며 권한 없이 자동으로 허용됩니다. 샌드박스화할 수 없는 명령어(허용되지 않은 호스트에 대한 네트워크 접근이 필요한 경우 등)는 일반 권한 흐름으로 대체됩니다. 명시적 거부 규칙은 항상 적용됩니다. 요청 규칙은 일반 권한 흐름으로 대체되는 명령어에만 적용됩니다.

**일반 권한 모드**: 모든 bash 명령어는 샌드박스화된 경우에도 표준 권한 흐름을 거칩니다. 더 많은 제어권을 제공하지만 더 많은 승인이 필요합니다.

두 모드 모두 샌드박스는 동일한 파일 시스템 및 네트워크 제한을 적용합니다. 차이점은 샌드박스화된 명령어가 자동 승인되는지 아니면 명시적 권한이 필요한지에 있습니다.

:::info
자동 허용 모드는 권한 모드 설정과 독립적으로 작동합니다. "편집 수락" 모드가 아니더라도 자동 허용이 활성화되면 샌드박스화된 bash 명령어는 자동으로 실행됩니다. 즉, 파일 편집 도구는 일반적으로 승인이 필요하더라도 샌드박스 경계 내에서 파일을 수정하는 bash 명령어는 프롬프트 없이 실행됩니다.
:::

### 샌드박싱 구성

`settings.json` 파일을 통해 샌드박스 동작을 사용자 정의할 수 있습니다. 전체 구성 참조는 [설정](/settings#sandbox-settings)을 참조하세요.

#### 특정 경로에 서브프로세스 쓰기 접근 권한 부여

기본적으로 샌드박스화된 명령어는 현재 작업 디렉토리에만 쓸 수 있습니다. `kubectl`, `terraform`, `npm` 같은 서브프로세스 명령어가 프로젝트 디렉토리 외부에 쓰기가 필요한 경우, `sandbox.filesystem.allowWrite`를 사용하여 특정 경로에 대한 접근 권한을 부여하세요:

```json
{
  "sandbox": {
    "enabled": true,
    "filesystem": {
      "allowWrite": ["~/.kube", "/tmp/build"]
    }
  }
}
```

이러한 경로는 OS 수준에서 적용되므로, 샌드박스 내에서 실행되는 모든 명령어(자식 프로세스 포함)가 이를 준수합니다. 이는 도구 전체를 `excludedCommands`로 샌드박스에서 제외하는 것보다 특정 위치에 대한 쓰기 접근이 필요한 도구에 권장되는 방식입니다.

`allowWrite`(또는 `denyWrite`/`denyRead`/`allowRead`)가 여러 [설정 범위](/settings#settings-precedence)에서 정의된 경우, 배열은 **병합**됩니다. 즉, 모든 범위의 경로가 대체되지 않고 결합됩니다. 예를 들어, 관리 설정에서 `/opt/company-tools`에 대한 쓰기를 허용하고 사용자가 개인 설정에 `~/.kube`를 추가하면 두 경로 모두 최종 샌드박스 구성에 포함됩니다. 이는 사용자와 프로젝트가 상위 우선순위 범위에서 설정된 경로를 복제하거나 재정의하지 않고 목록을 확장할 수 있음을 의미합니다.

경로 접두사는 경로 해석 방식을 제어합니다:

| 접두사 | 의미 | 예시 |
| :--- | :--- | :--- |
| `/` | 파일 시스템 루트로부터의 절대 경로 | `/tmp/build`는 `/tmp/build`로 유지 |
| `~/` | 홈 디렉토리 기준 상대 경로 | `~/.kube`는 `$HOME/.kube`가 됨 |
| `./` 또는 접두사 없음 | 프로젝트 설정의 경우 프로젝트 루트 기준, 사용자 설정의 경우 `~/.claude` 기준 상대 경로 | `.claude/settings.json`의 `./output`은 `<project-root>/output`으로 해석 |

절대 경로에 대한 이전의 `//path` 접두사도 여전히 작동합니다. 이전에 프로젝트 기준 해석을 기대하며 단일 슬래시 `/path`를 사용했다면 `./path`로 변경하세요. 이 구문은 절대 경로에 `//path`, 프로젝트 기준 경로에 `/path`를 사용하는 [읽기 및 편집 권한 규칙](/permissions#read-and-edit)과 다릅니다. 샌드박스 파일 시스템 경로는 표준 관례를 따릅니다: `/tmp/build`는 절대 경로입니다.

`sandbox.filesystem.denyWrite` 및 `sandbox.filesystem.denyRead`를 사용하여 쓰기 또는 읽기 접근을 거부할 수도 있습니다. 이들은 `Edit(...)` 및 `Read(...)` 권한 규칙의 경로와 병합됩니다. 거부된 영역 내 특정 경로의 읽기를 다시 허용하려면 `denyRead`보다 우선하는 `sandbox.filesystem.allowRead`를 사용하세요. 관리 설정에서 `allowManagedReadPathsOnly`가 활성화되면 관리 `allowRead` 항목만 적용되고 사용자, 프로젝트, 로컬 `allowRead` 항목은 무시됩니다. `denyRead`는 여전히 모든 소스에서 병합됩니다.

예를 들어, 현재 프로젝트에서의 읽기는 허용하면서 전체 홈 디렉토리에서의 읽기를 차단하려면 프로젝트의 `.claude/settings.json`에 다음을 추가하세요:

```json
{
  "sandbox": {
    "enabled": true,
    "filesystem": {
      "denyRead": ["~/"],
      "allowRead": ["."]
    }
  }
}
```

`allowRead`의 `.`는 이 구성이 프로젝트 설정에 있기 때문에 프로젝트 루트로 해석됩니다. 동일한 구성을 `~/.claude/settings.json`에 두면 `.`는 `~/.claude`로 해석되고, 프로젝트 파일은 `denyRead` 규칙에 의해 차단된 상태로 유지됩니다.

:::tip
모든 명령어가 기본적으로 샌드박싱과 호환되는 것은 아닙니다. 샌드박스를 최대한 활용하는 데 도움이 되는 몇 가지 참고 사항:

* 많은 CLI 도구는 특정 호스트에 접근해야 합니다. 이러한 도구를 사용하면 특정 호스트 접근 권한을 요청합니다. 권한을 부여하면 현재와 향후에 해당 호스트에 접근할 수 있게 되어 샌드박스 내에서 안전하게 실행됩니다.
* `watchman`은 샌드박스에서 실행과 호환되지 않습니다. `jest`를 실행하는 경우 `jest --no-watchman` 사용을 고려하세요.
* `docker`는 샌드박스에서 실행과 호환되지 않습니다. `excludedCommands`에 `docker *`를 지정하여 샌드박스 외부에서 강제 실행하는 것을 고려하세요.
:::

:::note
Claude Code에는 필요한 경우 샌드박스 외부에서 명령어를 실행할 수 있는 의도적인 탈출 경로 메커니즘이 포함되어 있습니다. 샌드박스 제한으로 인해 명령어가 실패하면(네트워크 연결 문제나 호환되지 않는 도구 등), Claude는 실패를 분석하고 `dangerouslyDisableSandbox` 매개변수로 명령어를 재시도할 수 있습니다. 이 매개변수를 사용하는 명령어는 실행하기 위해 사용자 권한이 필요한 일반 Claude Code 권한 흐름을 거칩니다. 이를 통해 Claude Code는 특정 도구나 네트워크 작업이 샌드박스 제약 내에서 작동할 수 없는 엣지 케이스를 처리할 수 있습니다.

[샌드박스 설정](/settings#sandbox-settings)에서 `"allowUnsandboxedCommands": false`로 설정하여 이 탈출 경로를 비활성화할 수 있습니다. 비활성화되면 `dangerouslyDisableSandbox` 매개변수는 완전히 무시되고 모든 명령어는 샌드박스 내에서 실행되거나 `excludedCommands`에 명시적으로 나열되어야 합니다.
:::

## 보안 이점

### 프롬프트 인젝션 방지

공격자가 프롬프트 인젝션을 통해 Claude Code의 동작을 성공적으로 조작하더라도, 샌드박스는 시스템을 안전하게 유지합니다:

**파일 시스템 보호:**

* `~/.bashrc`와 같은 중요한 설정 파일 수정 불가
* `/bin/`의 시스템 수준 파일 수정 불가
* [Claude 권한 설정](/permissions#manage-permissions)에서 거부된 파일 읽기 불가

**네트워크 보호:**

* 공격자가 제어하는 서버로 데이터 유출 불가
* 미승인 도메인에서 악성 스크립트 다운로드 불가
* 미승인 서비스에 예상치 못한 API 호출 불가
* 명시적으로 허용되지 않은 도메인 접근 불가

**모니터링 및 제어:**

* 샌드박스 외부의 모든 접근 시도는 OS 수준에서 차단됨
* 경계가 테스트될 때 즉각적인 알림 수신
* 거부, 일회성 허용, 또는 구성 영구 업데이트 중 선택 가능

### 공격 표면 축소

샌드박싱은 다음으로 인한 잠재적 피해를 제한합니다:

* **악성 의존성**: 유해한 코드가 포함된 NPM 패키지 또는 기타 의존성
* **침해된 스크립트**: 보안 취약점이 있는 빌드 스크립트 또는 도구
* **소셜 엔지니어링**: 사용자를 속여 위험한 명령어를 실행하게 하는 공격
* **프롬프트 인젝션**: Claude를 속여 위험한 명령어를 실행하게 하는 공격

### 투명한 운영

Claude Code가 샌드박스 외부의 네트워크 리소스에 접근하려고 할 때:

1. 작업이 OS 수준에서 차단됨
2. 즉각적인 알림 수신
3. 다음 중 선택 가능:
   * 요청 거부
   * 일회성 허용
   * 영구적으로 허용하도록 샌드박스 구성 업데이트

## 보안 제한 사항

* **네트워크 샌드박싱 제한**: 네트워크 필터링 시스템은 프로세스가 연결할 수 있는 도메인을 제한하여 작동합니다. 프록시를 통과하는 트래픽은 별도로 검사하지 않으며, 사용자는 정책에서 신뢰할 수 있는 도메인만 허용하도록 책임을 져야 합니다.

:::warning
`github.com`과 같이 광범위한 도메인을 허용하면 데이터 유출이 가능할 수 있다는 잠재적 위험에 주의해야 합니다. 또한 일부 경우에는 [도메인 프론팅](https://en.wikipedia.org/wiki/Domain_fronting)을 통해 네트워크 필터링을 우회할 수 있습니다.
:::

* **Unix 소켓을 통한 권한 에스컬레이션**: `allowUnixSockets` 구성은 샌드박스 우회로 이어질 수 있는 강력한 시스템 서비스에 대한 접근을 의도치 않게 부여할 수 있습니다. 예를 들어, `/var/run/docker.sock`에 대한 접근을 허용하면 docker 소켓 악용을 통해 호스트 시스템에 대한 접근을 효과적으로 부여하게 됩니다. 샌드박스를 통해 허용하는 unix 소켓을 신중하게 고려하는 것이 좋습니다.
* **파일 시스템 권한 에스컬레이션**: 지나치게 광범위한 파일 시스템 쓰기 권한은 권한 에스컬레이션 공격을 가능하게 할 수 있습니다. `$PATH`의 실행 파일이 포함된 디렉토리, 시스템 구성 디렉토리, 또는 사용자 쉘 구성 파일(`.bashrc`, `.zshrc`)에 대한 쓰기 허용은 다른 사용자나 시스템 프로세스가 이러한 파일에 접근할 때 다른 보안 컨텍스트에서의 코드 실행으로 이어질 수 있습니다.
* **Linux 샌드박스 강도**: Linux 구현은 강력한 파일 시스템 및 네트워크 격리를 제공하지만, 권한 있는 네임스페이스 없이 Docker 환경 내에서 작동할 수 있도록 하는 `enableWeakerNestedSandbox` 모드를 포함합니다. 이 옵션은 보안을 상당히 약화시키므로 다른 방식으로 추가 격리가 적용되는 경우에만 사용해야 합니다.

## 샌드박싱과 권한의 관계

샌드박싱과 [권한](/permissions)은 함께 작동하는 상호 보완적인 보안 계층입니다:

* **권한**은 Claude Code가 사용할 수 있는 도구를 제어하며 도구 실행 전에 평가됩니다. Bash, Read, Edit, WebFetch, MCP 및 기타 모든 도구에 적용됩니다.
* **샌드박싱**은 bash 명령어가 파일 시스템 및 네트워크 수준에서 접근할 수 있는 것을 제한하는 OS 수준 적용을 제공합니다. bash 명령어와 그 자식 프로세스에만 적용됩니다.

파일 시스템 및 네트워크 제한은 샌드박스 설정과 권한 규칙 모두를 통해 구성됩니다:

* `sandbox.filesystem.allowWrite`를 사용하여 작업 디렉토리 외부 경로에 서브프로세스 쓰기 접근 권한 부여
* `sandbox.filesystem.denyWrite` 및 `sandbox.filesystem.denyRead`를 사용하여 특정 경로에 대한 서브프로세스 접근 차단
* `sandbox.filesystem.allowRead`를 사용하여 `denyRead` 영역 내 특정 경로의 읽기 재허용
* `Read` 및 `Edit` 거부 규칙을 사용하여 특정 파일 또는 디렉토리에 대한 접근 차단
* `WebFetch` 허용/거부 규칙을 사용하여 도메인 접근 제어
* 샌드박스 `allowedDomains`를 사용하여 bash 명령어가 접근할 수 있는 도메인 제어

`sandbox.filesystem` 설정과 권한 규칙의 경로는 모두 최종 샌드박스 구성으로 병합됩니다.

이 [저장소](https://github.com/anthropics/claude-code/tree/main/examples/settings)에는 샌드박스 전용 예시를 포함한 일반적인 배포 시나리오를 위한 시작 설정 구성이 포함되어 있습니다. 이것을 시작점으로 사용하고 필요에 맞게 조정하세요.

## 고급 사용법

### 커스텀 프록시 구성

고급 네트워크 보안이 필요한 조직은 커스텀 프록시를 구현하여 다음을 수행할 수 있습니다:

* HTTPS 트래픽 복호화 및 검사
* 커스텀 필터링 규칙 적용
* 모든 네트워크 요청 로깅
* 기존 보안 인프라와 통합

```json
{
  "sandbox": {
    "network": {
      "httpProxyPort": 8080,
      "socksProxyPort": 8081
    }
  }
}
```

### 기존 보안 도구와의 통합

샌드박스 bash 도구는 다음과 함께 작동합니다:

* **권한 규칙**: 심층 방어를 위해 [권한 설정](/permissions)과 결합
* **개발 컨테이너**: 추가 격리를 위해 [devcontainer](/devcontainer)와 함께 사용
* **엔터프라이즈 정책**: [관리형 설정](/settings#settings-precedence)을 통해 샌드박스 구성 적용

## 모범 사례

1. **제한적으로 시작**: 최소한의 권한으로 시작하고 필요에 따라 확장
2. **로그 모니터링**: 샌드박스 위반 시도를 검토하여 Claude Code의 요구 사항 파악
3. **환경별 구성 사용**: 개발 및 프로덕션 컨텍스트에 따른 다른 샌드박스 규칙
4. **권한과 결합**: 포괄적인 보안을 위해 IAM 정책과 함께 샌드박싱 사용
5. **구성 테스트**: 샌드박스 설정이 정당한 워크플로우를 차단하지 않는지 확인

## 오픈 소스

샌드박스 런타임은 자체 에이전트 프로젝트에서 사용할 수 있는 오픈 소스 npm 패키지로 제공됩니다. 이를 통해 더 넓은 AI 에이전트 커뮤니티가 더 안전하고 보안성 높은 자율 시스템을 구축할 수 있습니다. 실행하려는 다른 프로그램을 샌드박스화하는 데도 사용할 수 있습니다. 예를 들어, MCP 서버를 샌드박스화하려면 다음을 실행하세요:

```bash
npx @anthropic-ai/sandbox-runtime <command-to-sandbox>
```

구현 세부 사항 및 소스 코드는 [GitHub 저장소](https://github.com/anthropic-experimental/sandbox-runtime)를 방문하세요.

## 제한 사항

* **성능 오버헤드**: 최소한이지만 일부 파일 시스템 작업이 약간 느릴 수 있음
* **호환성**: 특정 시스템 접근 패턴이 필요한 일부 도구는 구성 조정이 필요하거나 샌드박스 외부에서 실행해야 할 수 있음
* **플랫폼 지원**: macOS, Linux, WSL2 지원. WSL1은 지원되지 않음. 네이티브 Windows 지원은 계획 중

## 샌드박싱이 적용되지 않는 영역

샌드박스는 bash 서브프로세스를 격리합니다. 다른 도구는 다른 경계 하에서 작동합니다:

* **내장 파일 도구**: Read, Edit, Write는 샌드박스를 통하지 않고 권한 시스템을 직접 사용합니다. [권한](/permissions)을 참조하세요.
* **컴퓨터 사용**: Claude가 앱을 열고 화면을 제어할 때, 격리된 환경이 아닌 실제 데스크톱에서 실행됩니다. 앱별 권한 프롬프트가 각 애플리케이션을 제어합니다. [CLI에서 컴퓨터 사용](/computer-use) 또는 [데스크톱에서 컴퓨터 사용](/desktop#let-claude-use-your-computer)을 참조하세요.
